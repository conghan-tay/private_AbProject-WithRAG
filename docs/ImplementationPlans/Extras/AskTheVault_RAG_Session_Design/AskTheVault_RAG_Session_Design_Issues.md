Critical Issues
Browser WebSocket auth cannot use UserId header as written.
The plan says the browser sends UserId in the WS upgrade header ([AskTheVault_RAG_Session_Design.md (line 429)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:429)), but browser WebSocket APIs cannot set arbitrary headers. If the client is truly a browser, use a query param, cookie, or Sec-WebSocket-Protocol convention instead.

ASGI deployment is not actually specified against the current runtime.
The plan adds WS on :8001 ([plan (line 100)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:100)), but current compose exposes only 8000 ([docker-compose.yml (line 25)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docker-compose.yml:25)) and start.sh only starts Gunicorn WSGI ([start.sh (line 24)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/backend/start.sh:24)). The plan needs a concrete process model: separate rag_ws service is cleaner than running two servers in one container.

Ingest/decryption path is incompatible with current chunked encryption and references unless spelled out.
Current download resolves references first, then calls decrypt_file_stream(file, iv, size) ([views.py (line 141)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/backend/files/views.py:141), [encryption.py (line 60)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/backend/files/services/encryption.py:60)). The RAG plan says “read ciphertext → decrypt” generically. It must explicitly reuse the same storage-record resolution or selected duplicate/reference files will fail because reference rows have no file/IV.

No document text extraction strategy.
The vault stores arbitrary files: PDF, TXT, PNG, MP4, etc. The RAG plan jumps from decrypted bytes to chunking ([plan (line 139)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:139)). For PDFs/images/video, bytes are not usable retrieval text. The plan needs supported MIME types, extractor libraries, unsupported-file behavior, and tests.

No WebSocket/resource limits is too risky for this app.
The plan explicitly defers WS rate limiting ([plan (line 500)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:500)), but each session can trigger decrypt, extraction, embedding, Chroma memory growth, and external LLM cost. Add minimum limits: max selected files, max total bytes, max chunks, max question length, per-user active sessions, idle timeout, ask throttle, LLM timeout/cancel-on-disconnect.

Major Gaps
Local embedding model is unspecified. Pick the exact model/package, vector dimension, Docker dependency, model cache location, startup/download behavior, and memory budget. “Local model” is not decision-complete.
Chroma EphemeralClient needs hard verification. The plan should pin Chroma settings/version and explicitly disable persistence/telemetry if available. “Ephemeral” should be tested by checking no files are written.
State naming is inconsistent. The plan says connect moves to ready ([plan (line 132)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:132)), but the state table has connected before select ([plan (line 278)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:278)). Split states into connected_no_documents, ingesting, ready, answering.
Grounding is overclaimed. The plan says answers are grounded “only” in selected files, but later admits prompts are not enforcement ([plan (line 410)](/Users/conghantay/Desktop/Contract/dplat-file-vault-coding-challenge/docs/ImplementationPlans/Extras/AskTheVault_RAG_Session_Design.md:410)). Either soften the product claim or add post-generation verification/citation checks.
Tests are missing from the plan. Add Channels WebsocketCommunicator tests for auth, malformed frames, select once, reference-file ingest, cross-user skipping, unsupported MIME, no-answer short-circuit with LLM not called, streaming success, partial-stream failure, disconnect cleanup, and resource-limit rejection.
Overall: the architecture direction is sound, especially session-scoped ephemeral indexing. But before implementation, I’d revise the plan around browser-compatible auth, ASGI deployment, current encryption/reference semantics, text extraction, and minimum abuse controls.
