# Step 11: AskTheVault LLM Streaming Implementation

## Summary
Implement Step 11 from `AskTheVault_RAG_Session_Design_v02.md`: replace the Step 9 `done`-only placeholder with grounded LLM token streaming over the existing WebSocket protocol. Step 10’s streaming tests are already red for the expected reason: `AskVaultConsumer.stream_answer_tokens(...)` is missing.

## Key Changes
- Add `RAG_LLM_MODEL = os.environ.get("RAG_LLM_MODEL", "gpt-4.1-mini")` in `backend/core/settings.py`.
- Add `backend/files/services/rag_answer.py` for answer prompt and token streaming:
  - Build the exact v02 grounded system/user prompt from retrieved `Document.page_content` and `metadata["file_id"]`.
  - Use `langchain_openai.ChatOpenAI(model=settings.RAG_LLM_MODEL, temperature=0, streaming=True)`.
  - Return a sync iterator of non-empty token strings from `ChatOpenAI.stream(...)`.
- Update `backend/files/consumers.py`:
  - Add `AskVaultConsumer.stream_answer_tokens(self, question, retrieved_documents)` as the patchable Step 11 hook.
  - Update `generate_answer_messages(...)` to consume that sync iterator off the Channels event loop with `sync_to_async(..., thread_sensitive=True)`.
  - Emit one `{ "type": "token", "data": token }` per token.
  - On success, emit `{ "type": "done", "sources": sources }`.
  - On any LLM streaming exception, emit `{ "type": "error", "code": "llm_failed" }` and no `done`.
  - Preserve existing state reset to `ready` after success or failure.

## Public Contract
- Success stream:
  - `{ "type": "token", "data": "..." }`
  - `{ "type": "done", "sources": ["<file-id>", ...] }`
- LLM failure:
  - `{ "type": "error", "code": "llm_failed" }`
  - No terminal `done`.
- `done.sources` remains deterministic from retrieval metadata, not generated by the model.
- `no_answer` continues to short-circuit before any LLM call.

## Assumptions
- Use LangChain `ChatOpenAI` because v02 locks LangChain as the RAG framework and the repo already pins `langchain-openai`.
- Do not add conversation memory, persistent artifacts, redaction, frontend work, or new REST endpoints in this step.
- Default verification remains fake-based and does not require network access or a live OpenAI API key.
